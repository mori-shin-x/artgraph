# examples/kiro-integration — Kiro + artgraph

End-to-end walkthrough of `artgraph integrate kiro`: what the integration adds,
how Kiro's `requirements.md` / `design.md` / `tasks.md` map onto the artgraph
graph, and what `check` reports as you implement.

## How Kiro maps onto artgraph

| Kiro artifact                                | artgraph node / edge                                          |
| -------------------------------------------- | ------------------------------------------------------------- |
| `### Requirement N: …` heading in `requirements.md` | `req` node with ID `Requirement-N`                            |
| `design.md`, `tasks.md` siblings             | `doc` nodes; `design → requirements`, `tasks → design` via the **kiro convention** |
| `- [ ] N. Task …` line in `tasks.md`         | `task` node with ID `N`                                       |
| `_Requirements: 1, 2_` on a task             | planning-only `task → verifies` edges (not used for coverage) |
| `// @impl Requirement-1` in source           | `file → implements → Requirement-1` (the edge that **counts** for coverage) |
| `it("[Requirement-1] …")` in a test          | `test → verifies → Requirement-1`                             |

> **Important**: Kiro's `_Requirements: N_` IDs are bare numbers, while the
> heading parser produces `Requirement-N` IDs. They are **intentionally not
> reconciled** — task → req links from `_Requirements:` are treated as planning
> metadata. Coverage is driven by `@impl Requirement-N` tags in your source.

## Layout

```
examples/kiro-integration/
├── .artgraph.json
├── .kiro/specs/auth/
│   ├── requirements.md     # Requirement 1 / 2 (heading style)
│   ├── design.md           # auto-linked to requirements (kiro convention)
│   └── tasks.md            # 1, 2 with `_Requirements:` planning links
├── src/auth.ts             # `// @impl Requirement-1`
└── tests/auth.test.ts      # `it("[Requirement-1] …")`
```

`.kiro/steering/artgraph.md` is generated by `integrate kiro` and is
intentionally **not** committed — `examples/.gitignore` keeps it out so the
example always starts from a clean state.

## Step 1 — Scan as-is

```bash
cd examples/kiro-integration
node ../../dist/cli.js scan
```

```
Nodes: 9  Edges: 11
  req: 2  doc: 3  file: 1  test: 1  task: 2
```

```bash
node ../../dist/cli.js check
```

`Requirement-1` is fully implemented and verified; `Requirement-2` is still
on the to-do list:

```
UNCOVERED:
  Requirement-2
COVERAGE:
  Requirement-1: verified
  Requirement-2: untagged
```

This is the report Kiro's agent will see whenever it calls
`artgraph check --diff` after implementing.

## Step 2 — Install the steering file

`artgraph integrate kiro` writes a single steering file that teaches the Kiro
agent when to call `impact` / `check --diff` / `reconcile`:

```bash
node ../../dist/cli.js integrate kiro
```

```
✓ Integrated: kiro (Kiro)

Created (1):
  .kiro/steering/artgraph.md

Next:
  Restart Kiro so the new steering file is picked up.
  Run `artgraph scan && artgraph reconcile` to establish a fresh baseline.
```

Unlike Spec Kit, Kiro has no `extensions.yml` registry — the steering file
**is** the integration. Re-running `integrate kiro` is safe; pass `--force` to
overwrite a hand-edited steering file.

### Where the agent gets its cues

The generated `.kiro/steering/artgraph.md` tells the agent:

- **Before implementation** → `artgraph impact <path>` to scope affected reqs / docs.
- **After implementation** → `artgraph check --diff` to surface drift / orphans / uncovered.
- **On drift detection** → `artgraph reconcile` (only after human review).

## Uninstall

There is no CLI uninstall for Kiro — delete the steering file directly:

```bash
rm .kiro/steering/artgraph.md
```
